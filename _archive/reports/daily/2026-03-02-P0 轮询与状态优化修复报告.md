## 🔧 修复详情

### 1. 前端轮询间隔优化

#### 问题描述
- 前端在 init 阶段轮询间隔过短（250ms），导致：
  - 后端接口压力过大
  - 前端资源浪费
  - 用户体验下降（频繁请求但状态不变）

#### 修复方案
**文件**: `miniprogram/services/pollingManager.js`

**修改内容**:
```javascript
// 修复前
this.stageIntervals = {
  fast: 1000,     // 0-30%: 1 秒轮询一次
  medium: 2000,   // 30-60%: 2 秒轮询一次
  slow: 3000,     // 60-80%: 3 秒轮询一次
  final: 5000     // 80-100%: 5 秒轮询一次
};

// 修复后
this.stageIntervals = {
  init: 2000,     // init 阶段：2 秒轮询一次（修复：从 250ms 增加）
  fast: 1000,     // 0-30%: 1 秒轮询一次
  medium: 2000,   // 30-60%: 2 秒轮询一次
  slow: 3000,     // 60-80%: 3 秒轮询一次
  final: 5000     // 80-100%: 5 秒轮询一次
};
```

**进度计算逻辑优化**:
```javascript
_calculateProgressBasedInterval(progress) {
  // 【P0 修复】init 阶段（进度=0）使用 2 秒间隔
  if (progress === 0) {
    return this.stageIntervals.init;
  }
  
  // 其他阶段保持原有逻辑
  if (progress < this.progressThresholds.low) {
    return this.stageIntervals.fast + (progress / 30) * 500;
  }
  // ... 其他阶段逻辑
}
```

#### 修复效果
- ✅ init 阶段轮询间隔从 250ms 增加到 2000ms（**降低 87.5% 请求频率**）
- ✅ 减少后端接口压力
- ✅ 提升用户体验

### 2. 前端超时提示（30 秒无进度更新）

#### 问题描述
- 当 AI 平台响应缓慢时，前端长时间无进度更新
- 用户不知道发生了什么，以为系统卡死
- 缺乏用户友好的提示机制

#### 修复方案
**文件**: `miniprogram/services/pollingManager.js`

**新增字段**:
```javascript
this.lastProgressUpdateTimes = new Map();       // 最后进度更新时间记录
this.progressUpdateTimeout = 30000;             // 进度更新超时（30 秒）
```

**超时检测逻辑**:
```javascript
async _poll(executionId) {
  // 【P0 新增】检查是否 30 秒无进度更新
  if (this._isProgressUpdateTimeout(executionId)) {
    this._handleProgressUpdateTimeout(executionId);
    return;
  }
  
  // ... 正常轮询逻辑
  
  // 【P0 新增】如果进度有更新，记录更新时间
  const lastStatus = this.lastStatusMap.get(executionId);
  if (lastStatus && lastStatus.progress > 0) {
    this.lastProgressUpdateTimes.set(executionId, Date.now());
  }
}
```

**超时处理方法**:
```javascript
_handleProgressUpdateTimeout(executionId) {
  const elapsedSeconds = Math.floor(
    (Date.now() - this.lastProgressUpdateTimes.get(executionId)) / 1000
  );

  console.warn(
    `[PollingManager] ⚠️ 进度更新超时：${executionId}, ` +
    `已 ${elapsedSeconds}秒无进度更新`
  );

  // 显示用户提示
  wx.showModal({
    title: '诊断进度缓慢',
    content: `诊断任务已 ${elapsedSeconds}秒 没有进度更新，` +
             `可能是 AI 平台响应较慢，建议耐心等待或稍后重试。`,
    showCancel: false,
    confirmText: '继续等待'
  });

  // 重置最后更新时间，避免重复提示
  this.lastProgressUpdateTimes.set(executionId, Date.now());
}
```

#### 修复效果
- ✅ 30 秒无进度更新时自动提示用户
- ✅ 用户知道系统在运行，不是卡死
- ✅ 提升用户体验和透明度

### 3. 后端状态更新优化

#### 问题描述
- 前端一直看到 `stage='init'`，即使任务已经开始执行
- 后端状态更新不及时，导致前端无法正确显示进度
- 数据库和缓存状态不同步

#### 修复方案

##### 3.1 立即更新初始状态
**文件**: `backend_python/wechat_backend/nxm_execution_engine.py`

```python
# 【P0 颠覆性修复】立即更新 stage 为 ai_fetching，避免前端一直看到 init
# 借鉴 Google Cloud Run：任务启动后立即报告"运行中"状态
try:
    scheduler.update_progress(0, total_tasks, 'ai_fetching')
    api_logger.info(f"[NxM] ✅ 状态已更新为 ai_fetching")

    # 【P0 增强】同时写入数据库，确保状态立即持久化
    from wechat_backend.repositories.task_status_repository import save_task_status
    save_task_status(
        task_id=execution_id,
        stage='ai_fetching',
        progress=0,
        status_text='正在初始化 AI 连接',
        completed_count=0,
        total_count=total_tasks,
        is_completed=False
    )
    api_logger.info(f"[NxM] ✅ 数据库状态已同步：execution_id={execution_id}, stage=ai_fetching")
except Exception as e:
    api_logger.warning(f"[NxM] ⚠️ 初始状态更新失败：{e}")
```

##### 3.2 状态查询 API 优化
**文件**: `backend_python/wechat_backend/views/diagnosis_views.py`

```python
# 【P0 关键修复】确保 stage 和 status 字段及时更新
# 问题：前端一直看到 init 阶段，因为后端没有正确推导 stage
# 解决：根据 progress 和 is_completed 推导正确的 stage

stage = report_data.get('stage') or 'processing'
status = report_data.get('status') or 'processing'
progress = report_data.get('progress', 0)
is_completed = report_data.get('is_completed', False)

# 【P0 修复】如果 stage 是 init 但 progress > 0，强制推导
if stage == 'init' and progress > 0:
    api_logger.warning(
        f"[TaskStatus] ⚠️ 状态不一致：stage=init 但 progress={progress}%，" +
        f"强制推导为 ai_fetching"
    )
    stage = 'ai_fetching'

# 【P0 修复】如果 progress >= 100 但 stage 不是 completed，强制推导
if progress >= 100 and stage not in ['completed', 'failed']:
    api_logger.warning(
        f"[TaskStatus] ⚠️ 状态不一致：progress=100 但 stage={stage}，" +
        f"强制推导为 completed"
    )
    stage = 'completed'
    status = 'completed'
    is_completed = True

# 【P0 修复】如果 is_completed=True 但 stage 不是 completed，强制同步
if is_completed and stage not in ['completed', 'failed']:
    api_logger.warning(
        f"[TaskStatus] ⚠️ 状态不一致：is_completed=True 但 stage={stage}，" +
        f"强制同步为 completed"
    )
    stage = 'completed'
    status = 'completed'
```

## 🎯 架构师决策说明

### 决策 1: 轮询间隔调整
**依据**: Google Cloud Run / AWS Lambda 最佳实践  
**理由**: 
- init 阶段（0% 进度）通常是在等待后端任务启动
- 过于频繁的轮询不会加速任务执行，反而增加服务器压力
- 2 秒间隔既能及时响应，又不会造成资源浪费

### 决策 2: 30 秒超时提示
**依据**: UX 最佳实践 / 用户心理模型  
**理由**:
- 30 秒是用户等待的心理阈值（Nielsen Norman Group 研究）
- 超过 30 秒无反馈，用户会认为系统卡死
- 提示用户"继续等待"而不是强制停止，保持任务完整性

### 决策 3: 状态推导机制
**依据**: 分布式系统状态一致性模式  
**理由**:
- 后端状态可能因为并发、延迟等原因不一致
- 前端需要可靠的状态来判断是否停止轮询
- 通过多字段交叉验证（progress + stage + is_completed）确保准确性
