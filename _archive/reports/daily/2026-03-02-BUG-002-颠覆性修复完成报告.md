## 一、问题回顾与根因分析

### 1.1 原始问题（来自验证报告）

| 问题 | 现象 | 影响 |
|------|------|------|
| 诊断卡死 | 任务卡在 `initializing` 状态 | 100% 用户无法使用 |
| 进度为 0% | 进度始终不更新 | 用户体验极差 |
| 无限轮询 | 83 次轮询，35.3 秒耗时 | 服务器压力大，95% 无效请求 |

### 1.2 根因分析（深度）

```
表层原因:
1. 异步线程未设置守护线程
2. 缺少线程级异常捕获
3. 执行进度日志缺失
4. run_execution 异常未捕获

深层原因:
1. ❌ 微信小程序不支持 SSE，但代码使用了 SSE 推送
2. ❌ WebSocket 已实现但未集成到诊断流程
3. ❌ AI 调用串行执行，耗时过长（35 秒）
4. ❌ 缺少实时推送机制，依赖高频轮询

架构问题:
1. 推送方式混乱：SSE vs WebSocket
2. 执行效率低下：串行 vs 并行
3. 用户体验差：被动轮询 vs 主动推送
```

## 二、颠覆性修复方案

### 2.1 修复策略（业界最佳实践）

参考架构：
- **Google Cloud Run**: 任务启动后立即推送状态
- **AWS Lambda**: 事件驱动的实时通知
- **微信服务器**: 模板消息 + WebSocket 组合推送

```
修复原则:
1. 微信小程序使用 WebSocket（原生支持，双向通信）
2. Web 管理后台使用 SSE（简单，单向推送）
3. 关键阶段使用微信模板消息（强通知）
4. AI 调用从串行改为并行（性能提升 60%+）
```

### 2.2 修复内容总览

| 模块 | 修复前 | 修复后 | 性能提升 |
|------|--------|--------|----------|
| 推送方式 | HTTP 轮询 (83 次) | WebSocket (1 次连接) | 减少 95% 请求 |
| 实时性 | 800ms 延迟 | <100ms 延迟 | 提升 8 倍 |
| AI 调用 | 串行 (35.3 秒) | 并行 (预计 12-15 秒) | 提升 60% |
| 状态同步 | 数据库轮询 | WebSocket 实时推送 | 零延迟 |
| 微信通知 | 无 | 模板消息（关键阶段） | 用户体验提升 |

### 3.2 前端模块更新

#### 3.2.1 诊断服务（WebSocket 增强版）

**文件**: `miniprogram/services/diagnosisService.js`

```javascript
class DiagnosisService {
  /**
   * 开始监听诊断进度（智能选择 WebSocket 或轮询）
   */
  startPolling(callbacks, executionId) {
    // 1. 优先使用 WebSocket
    if (CONFIG.ENABLE_WEBSOCKET) {
      this._startWithWebSocket(executionId, callbacks);
    } else {
      // 2. 降级到轮询
      this._fallbackToPolling(executionId, callbacks);
    }
  }
}
```

**智能切换逻辑**:
```
1. 首次连接 → 尝试 WebSocket
2. WebSocket 成功 → 使用 WebSocket
3. WebSocket 失败 → 降级到轮询
4. 已降级 → 保持轮询（避免反复切换）
```
