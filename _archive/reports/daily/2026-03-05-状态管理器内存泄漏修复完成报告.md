## 一、问题描述

### 1.1 原问题

**修复前配置**:
- 清理间隔：1800 秒（30 分钟）
- 完成 TTL: 1800 秒（30 分钟）
- 无内存上限控制
- 无紧急清理机制

**问题表现**:
- 长时间运行后内存持续增长
- 已完成任务永久驻留内存
- 无自动清理或告警机制
- 可能导致 OOM（内存耗尽）崩溃

**影响范围**:
- 所有诊断任务执行
- 长时间运行的服务实例
- 高并发场景下尤为严重

**文件**: `state_manager.py:88-101`


### 2.3 实现紧急清理机制

**文件**: `state_manager.py:983-1001`

```python
# 紧急清理检查
current_size = len(self.execution_store)
is_emergency = current_size > self.max_memory_items

if is_emergency:
    self.emergency_cleanup_count += 1
    effective_ttl = self.completed_state_ttl_seconds / 2  # TTL 减半
    api_logger.warning(
        f"[StateManager] 🚨 内存超限触发紧急清理："
        f"current_size={current_size}, max_size={self.max_memory_items}, "
        f"effective_ttl={effective_ttl}s"
    )
```

**工作机制**:
1. 检测内存是否超限（>1000 项）
2. 超限时 TTL 减半（10 分钟 → 5 分钟）
3. 记录紧急清理次数用于告警
4. 快速释放内存

### 2.4 增强错误处理

**文件**: `state_manager.py:1003-1020`

```python
# 时间解析失败处理
except (ValueError, TypeError) as parse_error:
    api_logger.warning(
        f"[StateManager] ⚠️ 时间解析失败，保留数据：{execution_id}, "
        f"updated_at={updated_at_str}, error={parse_error}"
    )

# 清理失败详细日志

    api_logger.error(
        f"[StateManager] ❌ 清理失败 {execution_id}: {e}",
        exc_info=True
    )
```

**改进**:
- 区分不同类型错误
- 保留详细错误堆栈
- 失败时不影响其他任务清理
