## 二、详细修复方案

### 2.1 修复 1: Redis 客户端导入失败

**问题描述**:
```
[CacheWarmup] Redis 缓存失败：cannot import name 'get_redis_client' 
from 'wechat_backend.cache.redis_cache'
```

**根因分析**:
- `cache_warmup_tasks.py` 调用了 `get_redis_client()` 函数
- 但 `redis_cache.py` 只导出了 `get_redis_cache()` 和 `init_redis_cache()`
- 缺少 `get_redis_client()` 函数的导出

**修复方案**:

**文件**: `wechat_backend/cache/redis_cache.py`

```python
def get_redis_client():
    """
    获取 Redis 客户端实例（兼容旧代码）
    
    返回：
        Redis 客户端实例或 None（如果不可用）
    """
    cache_service = get_redis_cache()
    if cache_service and hasattr(cache_service, '_client'):
        return cache_service._client
    return None
```

**效果**:
- ✅ 兼容旧代码调用方式
- ✅ Redis 不可用时返回 None，系统自动降级到内存缓存
- ✅ 不阻塞启动流程

### 2.2 修复 2: 数据库索引创建超时

**问题描述**:
```
query_optimizer.py - init_recommended_indexes() - 表创建超时
检测到当前存在的表：[]
```

**矛盾点**:
- `init_db()` 日志显示"数据库表结构初始化完成"
- 但 `init_recommended_indexes()` 检测到表为空

**根因分析**:

1. **WAL 检查点未触发**: `init_db()` 使用直接连接创建表，但未触发 WAL 检查点
2. **连接可见性问题**: `init_recommended_indexes()` 使用连接池，可能看不到未检查点的更改
3. **时序问题**: 索引初始化可能在表创建事务完全提交前就开始执行

**修复方案**:

#### 修复 2.1: `database_core.py` - 确保 WAL 检查点

```python
conn.commit()

# P0-DB-INIT-003: 确保表创建后触发 WAL 检查点，使其他连接可见
conn.execute("PRAGMA wal_checkpoint(TRUNCATE)")

db_logger.info("数据库表结构初始化完成")
```

#### 修复 2.2: `query_optimizer.py` - 增强诊断和重试

**新增诊断日志**:
```python
api_logger.info(f"[索引初始化] 数据库路径：{index_manager.db_path}")
api_logger.info(f"[索引初始化] 数据库文件存在：{os.path.exists(index_manager.db_path)}")
api_logger.info(f"[索引初始化] 当前存在的表：{tables}")
api_logger.info(f"[索引初始化] 必需的表：{required_tables}")
api_logger.info(f"[索引初始化] 已存在的表：{existing_tables}")
```

**增强错误诊断**:
```python
if not tables:
    api_logger.warning(
        "[索引初始化] 数据库中没有任何表！"
        "这可能是 database_core.py:init_db() 未执行或数据库路径配置错误。"
    )

# 最终失败时的详细诊断
api_logger.error("可能的原因：1) init_db() 未执行 2) 数据库路径不一致 3) 权限问题")
if os.path.exists(index_manager.db_path):
    api_logger.error(f"数据库文件大小：{os.path.getsize(index_manager.db_path)} bytes")
else:
    api_logger.error(f"数据库文件不存在：{index_manager.db_path}")
```

**效果**:
- ✅ WAL 检查点确保事务对所有连接可见
- ✅ 详细诊断日志帮助快速定位问题
- ✅ 渐进式重试应对各种延迟场景

### 2.3 修复 3: AI 健康检查警告逻辑错误

**问题描述**:
```
✅ doubao: OK
✅ deepseek: OK
... (所有 7 个平台都 OK)
⚠️  AI Platform health check completed with warnings
```

**根因分析**:
- `platform_health_monitor.py:run_startup_health_check()` 第 347 行
- 无条件打印警告，即使所有平台都健康
- 返回的 `status` 字段可能导致 `app.py` 误判

**修复方案**:

#### 修复 3.1: `platform_health_monitor.py` - 条件化警告

```python
# 根据实际健康状态决定是否打印警告
if unhealthy_platforms:
    api_logger.warning(
        f"⚠️  {len(unhealthy_platforms)} platform(s) unhealthy: "
        f"{', '.join(unhealthy_platforms)}. System will proceed with reduced functionality."
    )
    api_logger.info(
        "AI Platform health check completed with warnings, "
        "system proceeding with available platforms."
    )
elif unconfigured_platforms:
    api_logger.info(
        "AI Platform health check completed. "
        "Some platforms not configured but system is functional."
    )
else:
    api_logger.info(
        "✅ AI Platform health check passed! All platforms are functional."
    )

return {
    'success': True,
    'status': HealthStatus.DEGRADED.value if unconfigured_platforms else HealthStatus.HEALTHY.value,
    # ...
}
```

#### 修复 3.2: `app.py` - 精确状态判断

```python
status = health_results.get('status', 'unknown')
unhealthy = health_results.get('unhealthy_platforms', [])

# 仅在不健康平台时记录错误
if status == 'unhealthy' and unhealthy:
    app_logger.error(f"⚠️  AI Platform health check FAILED! Unhealthy: {', '.join(unhealthy)}")
elif status == 'degraded':
    unconfigured = health_results.get('unconfigured_platforms', [])
    if unconfigured:
        app_logger.info(f"ℹ️  AI Platform health check: {len(unconfigured)} platform(s) not configured")
    else:
        app_logger.info("✅ AI Platform health check passed with minor warnings")
else:
    app_logger.info(f"✅ AI Platform health check passed! (status={status})")
```

**效果**:
- ✅ 仅在实际有问题时打印警告
- ✅ 区分"未配置"和"不健康"状态
- ✅ 所有平台健康时显示成功消息
