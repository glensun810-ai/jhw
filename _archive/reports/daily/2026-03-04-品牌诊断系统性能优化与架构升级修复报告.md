### 问题 1: 前端限流 - 防止并发请求

**问题描述**: 轮询逻辑中，setTimeout 在前一个请求返回之前可能触发下一个请求

**修复内容**:
- ✅ 在 `pollingManager.js` 中添加 `pendingRequests` Map
- ✅ 在 `_poll` 方法开始时检查是否有进行中的请求
- ✅ 使用 `try-finally` 确保请求锁正确释放

**修改文件**: `miniprogram/services/pollingManager.js`

**新增代码**:
```javascript
// 构造函数中
this.pendingRequests = new Map();  // 正在进行的请求

// _poll 方法中
async _poll(executionId) {
  // 【P0 修复 - 2026-03-04】检查是否有正在进行的请求
  if (this.pendingRequests.has(executionId)) {
    console.warn(`[PollingManager] ⚠️ 请求进行中，跳过轮询：${executionId}`);
    return;
  }

  try {
    // 【P0 修复 - 2026-03-04】设置请求锁
    this.pendingRequests.set(executionId, true);
    
    // ... 执行请求
    
  } finally {
    // 【P0 修复 - 2026-03-04】释放请求锁
    this.pendingRequests.delete(executionId);
  }
}
```

**修复效果**:
- ✅ 防止并发请求
- ✅ 避免请求堆积
- ✅ 减少服务器压力

**状态**: ✅ 已修复

### 问题 2: 前端指数退避

**问题描述**: 状态长时间不变时应拉长轮询间隔

**验证结果**:
- ✅ 检查 `pollingManager.js`，已存在完整的指数退避机制
- ✅ `_calculateProgressBasedInterval` 方法根据进度动态调整间隔
- ✅ `RetryStrategy.exponentialWithJitter` 提供指数退避 + 随机抖动

**现有实现**:
```javascript
// 根据进度动态调整轮询间隔
_calculateProgressBasedInterval(progress) {
  if (progress === 0) {
    return 2000;  // init 阶段：2 秒
  }
  
  if (progress < 30) {
    // 0-30%: 1-1.5 秒
    return 1000 + (progress / 30) * 500;
  } else if (progress < 60) {
    // 30-60%: 2-3 秒
    return 2000 + ((progress - 30) / 30) * 1000;
  } else if (progress < 80) {
    // 60-80%: 3-5 秒
    return 3000 + ((progress - 60) / 20) * 2000;
  } else {
    // 80-100%: 5-10 秒
    return 5000 + ((progress - 80) / 20) * 5000;
  }
}
```

**状态**: ✅ 已存在，无需修复

### 问题 5: 架构升级 - WebSocket 订阅模式

**问题描述**: 优先使用 WebSocket 推送，前端在 90% 后停止轮询，静默等待推送

**修复内容**:
- ✅ 在 `diagnosisService.js` 中添加 `_switchToWebSocketWait` 方法
- ✅ 在轮询回调中检测进度，到达 90% 后切换
- ✅ WebSocket 断开时自动降级回轮询

**修改文件**: `miniprogram/services/diagnosisService.js`

**新增代码**:
```javascript
_fallbackToPolling(executionId, callbacks) {
  pollingManager.startPolling(executionId, {
    onStatus: (status) => {
      // ...
      
      // 【P0 架构升级 - 2026-03-04】进度到达 90% 后，切换到 WebSocket 静默等待
      if (status.progress >= 90 && !this.isUsingWebSocket) {
        console.log(
          '[DiagnosisService] 📡 进度到达 90%，切换到 WebSocket 静默等待：' +
          `${executionId}`
        );
        this._switchToWebSocketWait(executionId, callbacks);
      }
    }
  });
}

_switchToWebSocketWait(executionId, callbacks) {
  // 停止轮询
  pollingManager.stopPolling(executionId);
  
  // 启动 WebSocket
  this.isUsingWebSocket = true;
  webSocketClient.connect(executionId, {
    onProgress: (data) => {
      if (callbacks.onStatus) {
        callbacks.onStatus({...data, mode: 'websocket'});
      }
      
      // 完成后停止
      if (data.progress >= 100 || data.should_stop_polling) {
        this._clearCurrentTask();
      }
    },
    onError: (error) => {
      // 降级回轮询
      this._fallbackToPolling(executionId, callbacks);
    }
  });
}
```

**工作流程**:
```
1. 诊断开始 → 使用 HTTP 轮询 (0-89%)
   └─ 轮询间隔：2 秒 → 10 秒（根据进度动态调整）

2. 进度到达 90% → 切换到 WebSocket
   └─ 停止轮询
   └─ 建立 WebSocket 连接
   └─ 静默等待推送

3. WebSocket 推送完成 (100%)
   └─ 调用 onComplete 回调
   └─ 清理任务

4. WebSocket 异常/断开
   └─ 降级回 HTTP 轮询
```

**优化效果**:
- ✅ 减少 90% 后的无效轮询
- ✅ 实时接收完成通知
- ✅ 降低服务器负载
- ✅ 提升用户体验

**状态**: ✅ 已实现
