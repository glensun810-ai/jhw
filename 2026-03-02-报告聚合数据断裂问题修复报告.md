# 2026-03-02 报告聚合阶段"数据断裂"问题修复报告

## 问题概述

前端日志显示，轮询在 90% (report_aggregating) 持续了很久，最后强行转为 completed。但随后请求 `/api/diagnosis/report/` 时后端报了 `'NoneType' object has no attribute 'get'` 错误。

## 根因分析

### 1. 核心瓶颈：报告聚合阶段的"数据断裂"

**现象**：
- 前端轮询卡在 90% (report_aggregating) 阶段
- 轮询超时后强行转为 completed
- 请求 `/api/diagnosis/report/` 时返回 500 错误

**诱因**：
- Redis 服务未运行，系统降级使用 MemoryCache
- Flask Debug 模式下运行多个进程
- 进程 A 负责执行 AI 任务并写入内存
- 进程 B 负责响应前端的 `/api/diagnosis/report/` 请求
- 结果：进程 B 的内存里根本没有进程 A 生成的中间数据，导致取出来是 None

**日志证据**：
```
[Redis] 连接被拒绝 (Error 61)
```

### 2. 前端逻辑 Bug：platform 变量未定义

**现象**：
```
ReferenceError: platform is not defined at reportAggregator.js:188
```

**原因**：
在 `calculateFirstMentionByPlatform` 函数中，`.map()` 回调里直接使用了 `platform` 变量，而没有从 `stats` 对象中提取。

**影响**：
即使后端修复了，前端的"战略看板"图表也依然会白屏。

### 3. 潜在的并发死锁风险

**日志证据**：
```
SQLITE_BUSY 风险
```

**分析**：
SQLite 在多线程写入时表现较弱。系统在同一秒内处理：
- 19 个索引创建
- 5 个缓存预热任务
- 3 个后台清理任务
- 前端每 500ms 一次的轮询压力

**后果**：
可能导致后端在尝试更新任务状态为 completed 时，数据库被锁定，导致写入的数据不完整或失败。

### Flask Debug 模式多进程问题

**问题描述**：
Flask 的 Debug 模式使用 Werkzeug reloader，会创建多个子进程。每个进程有独立的内存空间，导致：

1. 进程 A 执行 AI 任务，将结果写入 `MemoryCache A`
2. 进程 B 处理 `/api/diagnosis/report/` 请求，从 `MemoryCache B` 读取
3. `MemoryCache B` 中没有数据，返回 `None`
4. 触发 `'NoneType' object has no attribute 'get'` 错误

**解决方案**：
使用 Redis 作为共享缓存层，所有进程共享同一数据源。

### 后端防御性编程最佳实践

```python
# ❌ 错误示例（可能导致 NoneType 错误）
content = result.get('response').get('content')

# ✅ 正确示例（防御性编程）
response = result.get('response') or {}
content = response.get('content', "无返回内容")

# ✅ 更完善的示例（包含类型检查）
response = result.get('response', {})
if isinstance(response, dict):
    content = response.get('content', '')
    if not content or not content.strip():
        content = "AI 响应为空"
else:
    content = "AI 响应格式错误"
```
