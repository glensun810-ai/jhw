## 二、架构问题深度分析

### 2.1 当前架构设计

```
用户点击历史诊断
    ↓
history-detail.js onLoad
    ↓
loadDiagnosisResult(executionId) ← 从 Storage 加载
    ↓
✅ 缓存命中
    ↓
_immediateShowPage(localRecord)
    ↓
processHistoryDataOptimized(record)
    ↓
分层加载数据（4 层）
    ↓
展示页面
```

### 2.2 问题根因

#### 问题 1：缓存数据结构不匹配

**Storage 中存储的数据结构** (`storage-manager.js`):
```javascript
{
  executionId: "30a8af4b-943a-4c29-9886-728cc9d33e92",
  brandName: "趣车良品",
  completedAt: "2026-03-17T20:59:14.330003",
  timestamp: 1710706754330,
  version: "1.0.0",
  data: {
    results: [...],
    brandDistribution: {...},
    // ...
  }
}
```

**`loadDiagnosisResult` 返回的数据结构**:
```javascript
{
  executionId: "30a8af4b-943a-4c29-9886-728cc9d33e92",
  brandName: "趣车良品",
  completedAt: "2026-03-17T20:59:14.330003",
  timestamp: 1710706754330,
  version: "1.0.0",
  results: [...],           // ← 从 data 展开
  brandDistribution: {...}, // ← 从 data 展开
  // ...
}
```

**`processHistoryDataOptimized` 期望的数据结构**:
```javascript
{
  results: [...],           // ← 期望直接访问
  brandDistribution: {...}, // ← 期望直接访问
  // ...
}
```

**问题**: `processHistoryDataOptimized` 中提取 `results` 的逻辑：
```javascript
const results = record.results || record.result || null;
```

如果 `record.results` 存在但为空数组 `[]`，会被判断为"无数据"！

#### 问题 2：分层加载逻辑过于复杂

当前设计有 4 层延迟加载：
- 第 1 层：核心信息（0ms）
- 第 2 层：分析数据（100ms 延迟）
- 第 3 层：详细结果（200ms 延迟）
- 第 4 层：收藏状态（300ms 延迟）

**问题**:
1. 延迟加载导致数据展示慢
2. 每层都有默认值，导致"有数据但显示默认值"
3. 层与层之间数据依赖复杂

#### 问题 3：缓存加载和 API 加载混合

当前逻辑：
1. 先尝试从缓存加载
2. 缓存命中则立即展示
3. 同时发起 API 请求
4. API 返回后再次更新数据

**问题**:
- 缓存数据和 API 数据格式可能不一致
- 用户看到数据闪烁（缓存 → API）
- 增加复杂度

## 三、架构修复方案

### 3.1 核心原则

1. **单一数据源**: 历史诊断记录从数据库 API 加载，不使用缓存
2. **简单直接**: 减少分层加载，一次性展示所有数据
3. **错误处理**: 明确的错误提示，而非显示默认值

### 3.2 新架构设计

```
用户点击历史诊断
    ↓
history-detail.js onLoad
    ↓
调用 API: /api/diagnosis/report/{executionId}
    ↓
  ┌──────────────┬──────────────┐
  │              │              │
成功          失败 (404)    失败 (网络)
  │              │              │
  ↓              ↓              ↓
展示完整报告   提示"报告不存在"  提示"网络错误"
```

### 3.3 修复步骤

#### 步骤 1：移除缓存加载逻辑

**文件**: `brand_ai-seach/pages/history-detail/history-detail.js`

**修改 `onLoad` 方法**:

#### 步骤 2：简化 API 加载逻辑

**修改 `loadFromServer` 方法**:
```javascript
loadFromServer: function(executionId) {
  const serverUrl = getApp().globalData.serverUrl;

  wx.request({
    url: `${serverUrl}/api/diagnosis/report/${executionId}`,
    method: 'GET',
    timeout: 10000,
    success: (res) => {
      if (res.statusCode === 200 && res.data?.data) {
        // API 返回格式：{success: true, data: {...}}
        const report = res.data.data;
        this.displayReport(report);
      } else if (res.statusCode === 404) {
        console.error('[报告详情页] ❌ 报告不存在:', executionId);
        this.setData({ loading: false });
        wx.showModal({
          title: '报告不存在',
          content: '该诊断报告可能已被删除或从未生成',
          showCancel: false,
          success: () => wx.navigateBack()
        });
      } else {
        console.error('[报告详情页] ❌ 服务器错误:', res.statusCode);
        this.setData({ loading: false });
        wx.showModal({
          title: '加载失败',
          content: '无法加载报告数据，请稍后重试',
          showCancel: false
        });
      }
    },
    fail: (err) => {
      console.error('[报告详情页] ❌ 网络错误:', err);
      this.setData({ loading: false });
      wx.showModal({
        title: '网络错误',
        content: '无法连接服务器，请检查网络连接',
        showCancel: false
      });
    }
  });
},
```

## 四、缓存策略优化

### 4.1 缓存使用场景

**应该缓存**:
- ✅ 诊断完成后，跳转到报告页前的临时存储
- ✅ 网络不稳定时的降级方案

**不应该缓存**:
- ❌ 历史诊断记录的长期存储
- ❌ 作为主要数据源

### 4.2 新的缓存策略

**文件**: `brand_ai-seach/utils/storage-manager.js`

```javascript
/**
 * 保存诊断结果（仅用于临时缓存）
 * @param {string} executionId - 执行 ID
 * @param {Object} data - 诊断数据
 * @param {number} cacheTTL - 缓存时间（毫秒），默认 5 分钟
 */
function saveDiagnosisResult(executionId, data, cacheTTL = 5 * 60 * 1000) {
  try {
    const key = `diagnosis_result_${executionId}`;
    const storageData = {
      executionId,
      data,
      timestamp: Date.now(),
      cacheTTL  // 添加缓存过期时间
    };

    wx.setStorageSync(key, storageData);
    console.log(`[Storage] ✅ 诊断结果已缓存：${executionId} (有效期${cacheTTL/1000}秒)`);
  } catch (error) {
    console.error('[Storage] 保存诊断结果失败:', error);
  }
}

/**
 * 加载诊断结果（仅用于临时缓存）
 * @param {string} executionId - 执行 ID
 * @returns {Object|null} 诊断数据
 */
function loadDiagnosisResult(executionId) {
  try {
    const key = `diagnosis_result_${executionId}`;
    const storageData = wx.getStorageSync(key);

    if (!storageData) {
      return null;
    }

    // 检查缓存是否过期
    const now = Date.now();
    const cacheTTL = storageData.cacheTTL || 5 * 60 * 1000;
    if (now - storageData.timestamp > cacheTTL) {
      console.warn('[Storage] ⏰ 缓存已过期，清除');
      wx.removeStorageSync(key);
      return null;
    }

    return storageData.data;
  } catch (error) {
    console.error('[Storage] 加载诊断结果失败:', error);
    return null;
  }
}
```

## 六、总结

### 6.1 问题根因

1. **缓存数据结构不匹配** - Storage 返回的数据格式与期望不一致

2. **简化数据展示** - 一次性设置所有数据
3. **明确错误处理** - 404、网络错误等显示友好提示
