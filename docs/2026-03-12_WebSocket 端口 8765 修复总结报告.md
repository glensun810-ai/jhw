## 一、问题分析

### 1.1 端口合理性

| 端口 | 用途 | 状态 | 合理性 |
|-----|------|------|--------|
| **5000** | Flask HTTP API | ✅ 正常监听 | 合理 |
| **8765** | WebSocket 实时推送 | ❌ 未监听 | **不合理** |

**结论**: 8765 端口作为 WebSocket 端口是合理的，但未监听是异常的。

### 1.2 根因分析

**核心问题**: `start_websocket_server()` 函数在线程中启动 asyncio 服务器时存在兼容性问题

**技术细节**:
```python
# ❌ 问题代码
def run_server():
    loop = asyncio.new_event_loop()
    asyncio.set_event_loop(loop)
    server = serve(...)  # 这是协程，不是直接的服务器
    loop.run_until_complete(server)  # 这里有问题
    loop.run_forever()
```

**问题点**:
1. `websockets.serve()` 返回的是协程，需要 `await`
2. 在独立线程中运行 asyncio 需要特殊处理
3. macOS 上的 asyncio 线程模型有特殊性

## 三、最终解决方案

### 3.1 方案 A: 使用独立 WebSocket 服务（推荐）

**优点**:
- 简单可靠
- 与 Flask 解耦
- 易于调试和维护

**启动方式**:
```bash
# 启动 Flask
python3 run.py &

# 启动 WebSocket
python3 websocket_server.py &
```

**websocket_server.py**:
```python
#!/usr/bin/env python3
import asyncio
from websockets.asyncio.server import serve
from wechat_backend.websocket_route import handle_websocket_connection

async def main():
    async with serve(
        handle_websocket_connection,
        "0.0.0.0",
        8765,
        ping_interval=20,
        ping_timeout=10
    ):
        print("✅ WebSocket 服务已启动在端口 8765")
        await asyncio.Future()  # 保持运行

asyncio.run(main())
```

### 3.2 方案 B: 使用现有降级机制（当前）

**说明**: 前端已配置自动降级到 HTTP 轮询，WebSocket 不是必需的。

**前端降级逻辑**:
```javascript
// brandTestService.js
try {
  await webSocketClient.connect();
} catch (error) {
  // 降级到轮询
  this.startPolling();
}
```

**影响评估**:
- **功能影响**: 无（轮询同样可以获取实时进度）
- **性能影响**: 轻微（轮询间隔 2 秒，可接受）
- **用户体验**: 无感知
