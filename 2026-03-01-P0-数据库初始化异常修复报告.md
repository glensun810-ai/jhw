## 一、问题描述

### 1.1 症状

日志中出现大量警告：
```
2026-03-01 18:29:43,858 - wechat_backend.api - WARNING - query_optimizer.py:534 - init_recommended_indexes() - 表不存在，跳过索引创建：['users', 'test_records', 'brand_test_results', 'audit_logs', 'sync_results', 'cache_entries']
```

### 1.2 后果

- ❌ 诊断任务保存结果时崩溃（SQL Error: no such table）
- ❌ 前端无法读取任何历史报告
- ❌ 索引缺失导致查询性能下降

### 1.3 初步诊断

| 检查项 | 结果 |
|--------|------|
| 数据库文件权限 | ✅ 正常 |
| WAL 模式状态 | ✅ 正常工作 |
| 文件锁死 | ✅ 无锁死现象 |
| CREATE TABLE 语句 | ✅ 包含所有缺失表 |
| 多进程冲突 | ⚠️ **确认是根因** |

## 二、根因分析

### 2.1 Flask Debug 模式双进程问题

`.flaskenv` 配置：
```bash
FLASK_DEBUG=1
```

这启用了 Flask 的 **reloader 机制**，导致：

```
┌─────────────────────────────────────────────────────────┐
│                    Flask 启动流程                        │
├─────────────────────────────────────────────────────────┤
│  Monitor 进程 (监控文件变化)                              │
│       ↓                                                  │
│  启动 Worker 进程 (处理请求)                              │
│       ↓                                                  │
│  两个进程都会执行：                                       │
│    - init_db()                # 创建表                   │
│    - init_recommended_indexes()  # 创建索引              │
└─────────────────────────────────────────────────────────┘
```

### 2.2 时序竞争详解

```
时间轴:
│
├─ T0: Worker-1 启动，执行 init_db()
├─ T1: Worker-2 启动，执行 init_recommended_indexes()
├─ T2: Worker-2 调用 list_tables() → 表未创建完成
├─ T3: Worker-2 记录警告，跳过索引创建 ❌
├─ T4: Worker-1 完成表创建
│
└─ 结果：索引未创建，系统处于不一致状态
```

### 2.3 原重试机制不足

```python
# 原有代码 (query_optimizer.py:521-537)
max_retries = 3
retry_delay = 0.5  # 秒
# 总等待时间仅 1.5 秒，不足以应对多进程竞争
```

## 三、修复方案

### 3.1 方案 1：禁用 Reloader（生产环境推荐）

**文件**: `wechat_backend/app.py`

**修改**:
```python
# P0-DB-INIT-001: 禁用 reloader 避免双进程竞争
app.run(
    debug=Config.DEBUG,
    host='0.0.0.0',
    port=5000,
    use_reloader=False,  # 禁用 reloader
    threaded=True        # 启用多线程
)
```

**效果**:
- ✅ 保留 debug 调试功能
- ✅ 避免双进程竞争
- ✅ 支持多线程处理请求

### 3.2 方案 2：增强重试机制 + 进程锁（开发环境推荐）

**文件**: `wechat_backend/database/query_optimizer.py`

**核心改进**:

1. **文件锁防止多进程同时初始化**
```python
lock_file = '/tmp/wechat_db_init.lock'
lock_fd = open(lock_file, 'w')
fcntl.flock(lock_fd, fcntl.LOCK_EX)  # 独占锁
```

2. **渐进式重试策略**
```python
max_retries = 10
retry_delays = [0.5, 0.5, 1.0, 1.0, 2.0, 2.0, 3.0, 3.0, 5.0, 5.0]
# 总等待时间：20.5 秒
```

3. **WAL 检查点确保事务持久化**
```python
conn.execute("PRAGMA wal_checkpoint(PASSIVE)")
```

4. **双重验证确保表创建完成**
```python
# 第一次检查
if not missing_tables:
    time.sleep(0.5)  # 等待事务提交
    # 第二次验证
    verification_tables = index_manager.list_tables()
    if still_missing:
        continue  # 继续等待
```

5. **优雅降级：为已存在的表创建索引**
```python
# 即使部分表缺失，也为已存在的表创建索引
_create_indexes(index_manager, existing_tables)
```

## 八、总结

本次修复通过以下组合策略解决了数据库初始化异常问题：

1. **禁用 Reloader** - 从根源消除双进程竞争
2. **文件锁** - 确保单进程初始化
3. **渐进式重试** - 应对各种延迟场景
4. **WAL 检查点** - 确保事务持久化
5. **双重验证** - 防止竞态条件
6. **优雅降级** - 部分失败不影响整体

修复后，系统在各种场景下都能可靠地初始化数据库表结构和索引。
