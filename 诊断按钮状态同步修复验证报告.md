## 一、问题回顾

### 1.1 原问题描述

用户反馈：首页点击"启动诊断"之后，按钮始终停留在"诊断中"状态，无法正确同步后台诊断完成的状态。

### 1.2 根本原因分析

经过深入分析，发现以下核心问题：

1. **`parseTaskStatus` 函数优先使用 `stage` 字段而非 `status` 字段**
   - 原代码：`const status = statusData.stage || statusData.status;`
   - 问题：当后端返回 `status='completed'` 但 `stage='processing'` 时，前端无法识别完成状态

2. **完全忽略了 `should_stop_polling` 字段**
   - 后端明确返回 `should_stop_polling: true` 标记任务完成
   - 前端解析函数没有读取和使用这个最可靠的终止信号

3. **轮询终止逻辑未检查 `should_stop_polling`**
   - `brandTestService.js` 中的轮询只依赖 `isTerminalStatus()` 判断
   - 没有优先检查后端返回的 `should_stop_polling` 标志

if (backendShouldStopPolling) {

    parsed.is_completed = (statusData.status === 'completed');
    parsed.progress = 100;
  }
}

## 五、技术亮点

### 5.1 should_stop_polling 优先级最高

```
优先级判断顺序:
1. should_stop_polling (最高优先级 - 后端明确标记)
2. is_completed (次高优先级 - 后端完成标志)
3. status/stage 状态判断 (兼容旧逻辑)
```

### 5.2 status 字段优先于 stage

```
修复前：const status = statusData.stage || statusData.status;
修复后：const status = statusData.status || statusData.stage;

理由：
- status 是后端状态机明确标记的终态标识
- stage 可能只是阶段描述，不一定反映真实状态
- 例如：status='completed' 但 stage='processing' 时，应该以 status 为准
```

### 5.3 强制覆盖机制

```javascript
if (backendShouldStopPolling) {
  // 最可靠的终止信号，优先级高于所有其他判断
  if (statusData.status === 'completed' || statusData.status === 'failed') {
    parsed.stage = statusData.status;
    parsed.is_completed = (statusData.status === 'completed');
    parsed.progress = 100;
  }
}
```
